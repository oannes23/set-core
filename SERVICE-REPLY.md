## 1. (Blocking the daily flow) Who rolls the daily — and can it be *authored*?

The scheme is `seed = sha256("{date}|{rulesetVersion}|{contentVersion}")[:16]` with `specRef` as an
**opaque label** (`"daily/{date}/{ruleset}+{content}"`). That pushes the *selection* of which foe /
class / dungeon a day uses onto the **client**, deterministically derived from `seed` alone — the
server owns the seed, not the roll.

- `SERVICE.md` §2 read "server owns the daily-**roll** logic." Moving the roll client-side is
  defensible (still deterministic + version-pinned), but please make whose-job-it-is **explicit** so
  the client builds the right thing.
- The bigger issue: the original intent allowed dailies "generated by **or authored on** the server"
  — e.g. a hand-curated "Emberlord vs. a pyromancer" day. A pure hash seed **cannot express an
  authored daily**: there's no channel for the server to specify selections.

**Request:** add an **optional structured `spec` object** to the `/daily` descriptor now (content ids
+ params the client validates against its local registry), with the hash-seed path as the default
when `spec` is absent. Same "don't make phase-2 a schema break" instinct you applied to `actions` and
`manifestHash`. Confirm the daily-roll contract:
- **(a)** client derives all selections from `seed` (server stays seed-only), **or**
- **(b)** server can specify selections via a structured `spec` (authored dailies supported).

We'd prefer (b)'s *shape* shipped now even if MVP only exercises (a).

## 2. (Blocking the publish workflow) Where does the server get `rulesetVersion` / `contentVersion`?

Content authoring lives in **this** repo (`MODDING.md`); the server references content by id+version
only. Yet the server **advertises** current versions on `/health` and **folds them into the daily
seed** — so it must learn the "current official" versions from somewhere.

- How are they set — server config/env an operator bumps when new client content ships? An endpoint?
  This coupling point **is** our publish workflow, so we need it documented to build our side.
- Operational corollary: because the seed includes the versions, bumping them **mid-day silently
  re-rolls that date's board**. Acceptable if documented ("don't bump mid-day"); please state it.

## 3. (Confirm) Recovery when the token is lost but the fingerprint survives

Token + fingerprint share one client save key, so they're normally lost together and `/recover`
covers it. But a partial save-clear can drop the **token** while the **original fingerprint**
remains: `/register` would 409 ("already registered") and `/recover` reads as expecting a *new*
fingerprint (it 409s on "new fingerprint already registered").

**Confirm:** `/recover` re-issues a token for the identity **regardless of whether the presented
fingerprint is new or the original** (re-binding the original is a no-op + token re-issue). Otherwise
a player who loses only their token is locked out with no path back.

## 4. (Confirm) Which `/ingest` reject reasons are terminal vs retryable?

The client prunes the outbox on `accepted` and retries the remainder. `modded`,
`missing-version`, and `fingerprint-mismatch` look **permanent** — if the client retries them forever
the outbox never drains.

**Confirm** those three are **terminal** (the client will quarantine/drop, not re-send). If any
future reason is **transient** (a soft per-record reject the client *should* retry), please name it
distinctly so we can branch on retryable-vs-terminal rather than guessing from the string.

## 5. (Forward note, phase-2) `fastest-clear` is not replay-verifiable

`fewest-terms` and `deepest-delve` are derivable from a deterministic re-sim of the `actions` log;
`fastest-clear` (wall-clock `realTimeMs`) is **not**. When leaderboard anti-cheat lands, real-time
clears will need client timestamps inside the action stream plus a trust model, or a terms-based
speed metric instead. Nothing to change now — flagging the one criterion the replay substrate can't
cover so it isn't a surprise when boards arrive.

## 6. (Forward note) `consentVersion` bump behavior

`/health` advertises `consentVersion`, but it's undefined what happens when the consent text changes
for an already-consented player (re-prompt on next visit? grandfather?). Phase-2; a line in the spec
when convenient.
